# Feature Specification: Local Ollama Embedding Provider

**Feature Branch**: `002-ollama-embeddings`

**Created**: 2026-06-12

**Status**: Draft

**Input**: User description: "Replace the Voyage AI embedding provider with a local Ollama-based embedding provider to eliminate the external API dependency for local development."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Developer Runs Environment Without External Credentials (Priority: P1)

A developer clones the repository, runs `docker compose up`, and the full Tessera stack starts and processes documents without requiring any external API key or internet connectivity. Embeddings are generated by a locally-running service bundled with the compose stack.

**Why this priority**: This is the primary motivation for the feature. Without it, every developer must obtain and manage a paid API key before they can run even a basic local workflow. Removing this barrier is the core value proposition.

**Independent Test**: Can be verified by starting the compose stack on an air-gapped machine (or with network access blocked), ingesting a document, and confirming that chunks are embedded and searchable — all without any API key configured.

**Acceptance Scenarios**:

1. **Given** no `VOYAGE_API_KEY` is set, **When** a developer starts the compose stack, **Then** the stack starts successfully and the embedding service is available locally.
2. **Given** the compose stack is running, **When** a document is submitted for ingestion, **Then** the system generates embeddings without contacting any external service.
3. **Given** the compose stack is running with network access fully blocked, **When** a document is ingested, **Then** embeddings are generated and the document is searchable with no errors.

---

### User Story 2 - Existing Search Behaviour Is Preserved (Priority: P2)

A developer or end-user performs a semantic search query and receives ranked document results that are relevant to the query. The switch from the external provider to the local one does not degrade the ability to find and retrieve documents.

**Why this priority**: The embedding provider change must not break the product's core search capability. Similarity search must continue to work end-to-end; otherwise the switch causes a regression.

**Independent Test**: Can be tested by ingesting a known document set, executing a search query, and confirming that semantically relevant results are returned.

**Acceptance Scenarios**:

1. **Given** documents previously ingested are re-indexed after migration, **When** a user submits a semantic search query, **Then** relevant results are returned.
2. **Given** the local embedding service is unavailable (crashed), **When** an ingestion or search is attempted, **Then** the system returns a clear error message and does not silently return incorrect results.

---

### User Story 3 - Existing Vector Data Is Migrated (Priority: P3)

Any previously stored vector data is handled during the upgrade: either migrated or cleared and re-indexed, so the system remains internally consistent with the new embedding dimensions and no stale data causes silent errors.

**Why this priority**: Without migration, old vectors with incompatible dimensions would coexist with new vectors, causing undefined search behaviour. Consistency is required for correctness.

**Independent Test**: Can be tested by running the migration on a database containing old vectors, then performing a search, confirming results are returned without dimension-mismatch errors.

**Acceptance Scenarios**:

1. **Given** an existing deployment has stored vectors under the old dimension schema, **When** the migration is applied, **Then** the vector column is updated to the new dimension and data is either migrated or cleared for re-indexing.
2. **Given** the migration has been applied, **When** a new document is ingested, **Then** the embedding is stored without schema errors.

---

### Edge Cases

- What happens when the local embedding service is unreachable at ingestion time?
- How does the system handle a partial migration (upgrade interrupted mid-flight)?
- What happens to existing stored vectors that were generated with the old dimension — are they deleted, migrated, or flagged for re-indexing?
- How does the system behave when the embedding model has not yet been pulled by the local service (first cold start)?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST generate embeddings using a locally-running embedding service without requiring any external API key or internet access at runtime.
- **FR-002**: The local embedding service MUST be declared as a named service in the compose infrastructure definition and started automatically with the rest of the stack.
- **FR-003**: The configuration MUST accept a base URL for the local embedding service, defaulting to the service name declared in the compose stack.
- **FR-004**: The `VOYAGE_API_KEY` environment variable MUST be removed from all compose service definitions and from application configuration.
- **FR-005**: The existing embedding provider port interface MUST remain unchanged; only the adapter implementation is replaced.
- **FR-006**: The system MUST apply a database schema migration that updates the vector column to the dimension count required by the new embedding model.
- **FR-007**: The vector store MUST remain the sole system of record for all persistent embedding data; no secondary store is introduced.
- **FR-008**: The previous external-API embedding adapter MUST be deleted from the codebase; it MUST NOT be left as dead code or an optional path.
- **FR-009**: The system MUST emit a clear, structured error when the local embedding service is unreachable, rather than silently failing or returning empty results.
- **FR-010**: The compose infrastructure definition MUST declare the embedding model to be used, so that pulling images and model weights requires no manual steps beyond `docker compose pull`.

### Key Entities *(include if feature involves data)*

- **Embedding Provider (Port)**: The abstract interface that the rest of the system calls to convert text into a vector. This interface is unchanged by this feature.
- **Ollama Embedding Adapter**: The new concrete implementation of the Embedding Provider port. It calls the local Ollama HTTP API and returns a vector of the appropriate dimension.
- **Embedding Vector**: A fixed-length numerical representation of a text chunk. The dimension changes from 1024 to 768 as part of this migration.
- **Vector Column (Database)**: The column in the persistent store that holds embedding vectors for each document chunk. Its dimension constraint must be updated via a migration.
- **Local Embedding Service**: The containerised service that hosts the embedding model and exposes a local HTTP API. It is declared in the compose infrastructure and has no external network dependency at runtime.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can start the full compose stack and ingest a document end-to-end without setting any API key — measured by executing a full ingest on a clean environment with no credential configuration.
- **SC-002**: The compose stack reaches a healthy ready state on an air-gapped network (no outbound internet) within the same time window as the current connected startup, confirming zero external calls at runtime.
- **SC-003**: Semantic search returns relevant results for a test query after migration, confirming functional equivalence with the previous provider for core retrieval use cases.
- **SC-004**: Zero references to the removed external embedding provider remain in the codebase after the change, confirmed by automated search of the source tree.
- **SC-005**: The database migration applies cleanly on both a fresh schema and an existing database with old vector data, with no manual intervention required.

## Assumptions

- Developers are expected to have Docker and Docker Compose available; no additional tooling is required to use the local embedding service.
- The local embedding model (`nomic-embed-text`, 768 dimensions) produces embeddings of sufficient quality for the retrieval tasks Tessera supports; no formal quality benchmark is required as a gate for this feature.
- Existing stored vectors will be deleted and documents re-indexed rather than numerically transformed, since dimension-changing transformations are not semantically valid.
- The local embedding service image and model weights are fetched during `docker compose pull`; a working internet connection is required only at image/model pull time, not at runtime.
- This feature targets the local development and self-hosted deployment path; a production deployment using a managed embedding service is out of scope for this specification.
- The port interface contract (method signatures and return types) is stable and will not require changes to accommodate the new adapter.
