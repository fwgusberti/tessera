# Implementation Plan: Document Reindex UI

**Branch**: `017-document-reindex-ui` | **Date**: 2026-06-20 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/017-document-reindex-ui/spec.md`

## Summary

Add a "Reindex" button to the document detail page (`/documents/[id]`) that calls the existing `POST /v1/documents/{document_id}/reindex` endpoint. The button is visible only to the document owner or an admin, and only when the document is in `published` state. On success it shows "Reindex queued" for 3 seconds then re-enables; on failure it shows the server error inline. No new routes, no new backend changes, no schema changes.

#### 1c. Add reindex state fields

```ts
const [reindexing, setReindexing] = useState(false);
const [reindexMessage, setReindexMessage] = useState<string | null>(null);
const [reindexError, setReindexError] = useState<string | null>(null);
```

#### 1d. Add `handleReindex` function

Place alongside the existing `handlePublish` function:

```ts
const handleReindex = async () => {
  if (!document) return;
  setReindexing(true);
  setReindexError(null);
  setReindexMessage(null);
  try {
    await api.post(`/v1/documents/${id}/reindex`, {});
    setReindexMessage("Reindex queued");
    const timer = setTimeout(() => {
      setReindexMessage(null);
      setReindexing(false);
    }, 3000);
    return () => clearTimeout(timer);
  } catch (err: unknown) {
    setReindexError(err instanceof Error ? err.message : "Failed to queue reindex");
    setReindexing(false);
  }
};
```

Note: The `clearTimeout` cleanup prevents stale state updates if the user navigates away before the 3-second window closes. In a `handleReindex` function (not a `useEffect`), store the timer in a `useRef` so it can be cleaned up; alternatively, wrap the post-success timer in a `useEffect` keyed on `reindexMessage`. The simplest correct pattern for this codebase (matching existing style) is to store the timer ref:

```ts
const reindexTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const handleReindex = async () => {
  if (!document) return;
  if (reindexTimerRef.current) clearTimeout(reindexTimerRef.current);
  setReindexing(true);
  setReindexError(null);
  setReindexMessage(null);
  try {
    await api.post(`/v1/documents/${id}/reindex`, {});
    setReindexMessage("Reindex queued");
    reindexTimerRef.current = setTimeout(() => {
      setReindexMessage(null);
      setReindexing(false);
    }, 3000);
  } catch (err: unknown) {
    setReindexError(err instanceof Error ? err.message : "Failed to queue reindex");
    setReindexing(false);
  }
};
```

#### 1e. Add visibility derivation

```ts
const canReindex =
  document !== null &&
  document.state === "published" &&
  (user?.id === document.owner_user_id || user?.isAdmin === true);
```

Place this inside the render body, after `document` is confirmed to be non-null.

#### 1f. Add Reindex button JSX

In the header actions area, after (or in place of) the `document.state === "ingested"` Publish button block:

```tsx
{canReindex && (
  <div className="flex flex-col items-end gap-1">
    <button
      onClick={handleReindex}
      disabled={reindexing}
      className="bg-blue-600 text-white px-4 py-2 rounded text-sm font-medium hover:bg-blue-700 disabled:opacity-50"
    >
      {reindexing ? "Reindexing…" : "Reindex"}
    </button>
    {reindexMessage && (
      <p className="text-xs text-green-600">{reindexMessage}</p>
    )}
    {reindexError && (
      <p className="text-xs text-red-600">{reindexError}</p>
    )}
  </div>
)}
```

The Publish button block and the Reindex button block are **siblings** in the JSX — they are not nested. Since the Publish button only renders for `state === "ingested"` and Reindex only for `state === "published"`, they are mutually exclusive.
